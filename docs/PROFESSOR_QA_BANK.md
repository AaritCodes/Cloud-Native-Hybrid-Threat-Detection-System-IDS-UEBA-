# Professor Viva Q and A Bank (AI + Technical)

This bank is built from the current codebase implementation, not only from presentation claims.
Use it as a rehearsal document before demo day.

---

## 0) One-Minute Technical Opening

Q: Explain your system in one minute.
A: My system has two detection engines and one response layer. IDS reads CloudWatch network metrics and uses a RandomForest model to produce network risk. UEBA reads CloudTrail logs from S3 and uses Isolation Forest to produce user risk. I fuse both risks using Final Risk = 0.6 x Network Risk + 0.4 x User Risk. Based on thresholds, the autonomous response layer logs, alerts, rate-limits, or blocks. In the agentic variant, a ReAct-style LLM agent adds tool-based reasoning, stores decisions in SQLite, and supports feedback-driven learning, with fallback to rule-based decisions if AI is unavailable.

---

## 1) Architecture and Core Flow

1. Q: What are the main runtime components?
   A: IDS engine, UEBA engine, fusion engine, alert system, and autonomous response agent.

2. Q: How do IDS and UEBA interact?
   A: They run independently each cycle. Then their scores are fused into one final risk score.

3. Q: What exactly is your fusion formula?
   A: Final Risk = (0.6 x Network Risk) + (0.4 x User Risk).

4. Q: Why is network weighted higher than user behavior?
   A: For external attacks like DDoS, network anomalies appear first and are stronger signals. UEBA adds context and false-positive control.

5. Q: What are your action thresholds?
   A: Implemented thresholds are: final_risk > 0.8 CRITICAL, > 0.6 HIGH, > 0.4 MEDIUM, else LOW.

6. Q: What actions map to these levels?
   A: LOW -> LOG, MEDIUM -> ALERT, HIGH -> RATE_LIMIT, CRITICAL -> BLOCK.

7. Q: How often does the system run?
   A: Baseline runner sleeps 10 seconds between cycles. Agentic runner currently sleeps 15 seconds.

8. Q: What AWS region and target instance are configured?
   A: Region is ap-south-1 and IDS is configured for a specific EC2 instance ID in code.

9. Q: What happens if UEBA has no matching user signal for an IDS event?
   A: User risk defaults to 0.1 for fusion.

10. Q: Is the dashboard tightly coupled with detection engines?
	A: The newer web dashboard is a thin client that reads web_state.json generated by the main agentic process.

11. Q: How is historical state persisted?
	A: Decisions and IP history are persisted to SQLite in logs/decisions.db.

12. Q: What is your graceful-failure strategy?
	A: If Ollama/LLM is unavailable, decisions fall back to deterministic threshold rules so detection still continues.

## 2) ML Models and Feature Engineering

13. Q: Which model is used for IDS in your actual implementation?
	A: RandomForestClassifier loaded from models/ddos_model.pkl.

14. Q: Which model is used for UEBA?
	A: IsolationForest loaded from models/uba_model.pkl.

15. Q: How many features does IDS use at runtime?
	A: 6 features.

16. Q: Name IDS features.
	A: network_in, packets_in, bytes_per_packet, packet_rate, byte_rate, traffic_intensity.

17. Q: What are IDS key model parameters?
	A: n_estimators=100, random_state=42, default RandomForest settings otherwise.

18. Q: How many UEBA features are used?
	A: 4 features.

19. Q: Name UEBA features.
	A: hour, day, activity_volume, service_diversity.

20. Q: What is UEBA contamination?
	A: The persisted model currently has contamination=0.05.

21. Q: How does UEBA convert anomaly scores into risk?
	A: It computes decision_function, then min-max normalizes and inverts to 0-1 user_risk.

22. Q: What if UEBA scores are flat (max == min)?
	A: user_risk is set to 0.1.

23. Q: Why not deep learning?
	A: Feature space is low dimensional and tabular. Tree methods are faster, simpler to explain, and cheaper for this use case.

24. Q: What does IDS do to catch extreme spikes even if model confidence is uncertain?
	A: It applies rule-based safety boosts: >8M bytes or >15k packets forces high risk, and >4M or >8k forces elevated risk.

25. Q: How is risk produced from RandomForest output?
	A: From predict_proba probabilities, then mapped to a bounded risk scale with additional rule-based boosting.

26. Q: Is your IDS unsupervised?
	A: No. IDS is supervised RandomForest in the current code. UEBA is unsupervised Isolation Forest.

27. Q: Why might a professor see docs saying Isolation Forest for IDS?
	A: Some documentation reflects earlier design; current code and model artifact use RandomForest for IDS.

## 3) Agentic AI Design (ReAct + Tools + Memory)

28. Q: What makes this system agentic, not just LLM-wrapped?
	A: It follows an Observe -> Think -> Tool Call -> Decide -> Reflect loop with persistent memory and feedback recording.

29. Q: Which class implements the ReAct flow?
	A: AgenticThreatAgent.

30. Q: What is the max number of tool calls per decision?
	A: 3 tool calls per reasoning cycle.

31. Q: What model is preferred by the agent?
	A: qwen2.5:0.5b by default, with automatic resolution to available local Ollama models.

32. Q: What is the agent temperature and why?
	A: 0.3 for more stable decisions while still allowing some reasoning flexibility.

33. Q: What timeout is used for LLM generation calls?
	A: 180 seconds.

34. Q: What tools can the agent call?
	A: Internal tools include check_ip_reputation, get_attack_history, get_similar_threats, get_network_baseline, correlate_recent_events, get_accuracy_stats, get_adaptive_thresholds, and optional check_ip_external.

35. Q: How does the agent parse tool calls?
	A: It regex-parses function-like patterns and validates tool names against a registry.

36. Q: How is final action parsed from LLM output?
	A: Structured fields are extracted (ACTION, CONFIDENCE, RISK_LEVEL, REASONING), validated, and defaulted if malformed.

37. Q: How is memory persisted?
	A: Every decision is written to SQLite with risks, action, confidence, reasoning, model, context JSON, and optional outcome labels.

38. Q: What outcomes can be recorded for learning?
	A: true_positive, false_positive, missed_attack, benign.

39. Q: How are adaptive thresholds suggested?
	A: From historical false-positive and missed-attack rates after minimum evaluated outcomes; suggestions are bounded within safe ranges.

40. Q: Are adaptive thresholds auto-applied in runtime actions?
	A: Not yet. The current system computes and exposes suggested thresholds; automatic policy update is a next step.

41. Q: How do you prevent runaway tool usage?
	A: Hard cap of 3 tool calls per cycle.

42. Q: How do you prevent total failure when AI is down?
	A: Immediate rule-based fallback path with deterministic threshold mapping.

43. Q: What is stored for explainability?
	A: Reasoning text, action, confidence, model, tools used, and full decision context metadata.

44. Q: Is there any external threat intel tool?
	A: Optional AbuseIPDB lookup exists, enabled only when API key is present.

45. Q: What is your privacy stance for AI?
	A: Primary AI path runs on local Ollama, avoiding third-party prompt upload.

## 4) Data Pipeline and Feature Inputs

46. Q: What data source powers IDS?
	A: CloudWatch metrics NetworkIn and NetworkPacketsIn.

47. Q: What data source powers UEBA?
	A: CloudTrail logs fetched from S3 in .json.gz format.

48. Q: What CloudWatch aggregation is used?
	A: 5-minute window with 300-second period and Sum/Average stats.

49. Q: What if CloudWatch returns no datapoints?
	A: IDS falls back to baseline values (15,000 bytes and 72 packets) rather than zero.

50. Q: How many CloudTrail files are read each cycle?
	A: Up to 5 latest objects under today’s prefix.

51. Q: Why could UEBA miss some events?
	A: It intentionally limits files for speed and CloudTrail itself has ingestion delay.

52. Q: How are missing user IDs handled in diagnostics script?
	A: Missing user values are replaced with source IP to preserve grouping.

53. Q: Do you have local demo datasets in the repo?
	A: Yes, demo_materials/ddos_dataset.csv and demo_materials/parsed_logs.csv.

54. Q: What are row counts of demo datasets?
	A: 35,835 rows for ddos_dataset.csv and 570 rows for parsed_logs.csv.

55. Q: Are these demo datasets your training datasets?
	A: Not exactly. They are primarily used for demo/testing and diagnostics with proxy labels.

56. Q: What labels are used in diagnostics for IDS?
	A: VPC-style action proxy: REJECT=1, ACCEPT=0.

57. Q: What labels are used in diagnostics for UEBA?
	A: Public IPv4 heuristic from source_ip acts as suspicious label.

## 5) Evaluation, Metrics, and Scientific Defense

58. Q: Your diagnostics AUC looks low. Why?
	A: Because diagnostics use proxy labels and distribution-mismatched demo data, not necessarily the same regime as model training. The script is transparent about this.

59. Q: What are the latest diagnostics AUC values in artifacts?
	A: IDS about 0.50 and UEBA about 0.24 in current metrics_summary.json.

60. Q: How do you defend this in viva?
	A: Clarify that these are sanity diagnostics on proxy-labeled data. Operational demo claims are based on controlled attack simulation and response behavior, not benchmark-grade supervised evaluation.

61. Q: Why not report only accuracy?
	A: Class imbalance and proxy labels can make accuracy misleading; ROC/AUC and confusion matrix reveal model behavior more honestly.

62. Q: What is your strongest empirical claim?
	A: In controlled demo conditions, the system detects significant attack traffic quickly and triggers graduated response logic.

63. Q: What caveat must you always add with 0 percent false positives?
	A: It was achieved in controlled scenarios, not guaranteed for unconstrained production traffic.

64. Q: What is your ablation-style argument for the AI layer?
	A: Detection decision still exists without AI. AI adds reasoning, tool-driven context gathering, and memory-based justification.

65. Q: How can you improve experimental rigor next?
	A: Add labeled datasets, temporal holdout evaluation, drift tests, and production-like replay benchmarks.

66. Q: What versioning risk did you observe in model loading?
	A: InconsistentVersionWarning from scikit-learn indicates model was pickled under a different sklearn version than runtime.

67. Q: Why is that important?
	A: It can cause subtle behavior drift or incompatibility; production should pin versions or retrain and reserialize.

## 6) Security and Autonomous Response Questions

68. Q: How does autonomous blocking work in code?
	A: The agent attempts to modify EC2 security group ingress rules based on critical risk.

69. Q: Tough challenge: security groups do not support explicit deny. How do you answer?
	A: Correct. Security groups are allow-based. For production-grade blocking, use WAF/NACL deny controls or remove offending allow paths. Current code demonstrates automated control-plane action flow.

70. Q: What is block timeout policy?
	A: Default 10 minutes, then automatic unblock check runs periodically.

71. Q: What is rate-limit action in current implementation?
	A: It is a simulated rate-limit action plus alert; production should bind this to WAF/API Gateway throttling.

72. Q: How many alerts can email send per hour?
	A: Max 10 emails/hour as rate-limit guard.

73. Q: Is email credential handling production-safe?
	A: No. Current config stores credentials in JSON for demo convenience; production should use Secrets Manager or environment-injected secrets.

74. Q: What IAM permission is critical for autonomous response?
	A: EC2 security group ingress modification permissions (authorize/revoke).

75. Q: What happens if AWS API call fails during block?
	A: The function catches and logs error, returns BLOCK_FAILED or safe failure path.

76. Q: Can this design create false blocks?
	A: Yes in principle, so bounded timeout, human feedback loop, and conservative policy are essential.

77. Q: Is this safe for public internet production as-is?
	A: Not fully. It is a strong prototype; production needs hardened policy engine, deny-capable controls, and extensive validation.

78. Q: Does the current IDS identify attacker IP directly?
	A: IDS output currently uses an instance-level placeholder identifier; true source-IP blocking requires direct source attribution from flow logs/WAF pipeline.

79. Q: Why is that important?
	A: Response quality depends on accurate threat source identity, not only aggregate anomaly signals.

## 7) N8N, RAG, and LangFuse Integration Questions

80. Q: Are N8N, RAG, and LangFuse integrated in code?
	A: Yes, through IntegrationPipeline and supporting modules.

81. Q: Are those integrations active by default in main detection runner?
	A: Not in the core baseline loop by default; integration pipeline is available and can be wired in.

82. Q: What does N8N receive?
	A: Structured webhook payloads containing threat level, risk scores, traffic stats, and optional AI context.

83. Q: What does RAG contribute?
	A: It retrieves threat-intel context chunks from a local Chroma vector store and injects them into the LLM prompt.

84. Q: What happens if LangChain/Chroma is not installed?
	A: RAG gracefully disables and returns empty context without breaking the pipeline.

85. Q: What does LangFuse contribute?
	A: End-to-end LLM trace spans, prompt/response observability, latency and scoring hooks.

86. Q: How is decision quality fed back to LangFuse?
	A: Through score_decision style API with outcome labels and numeric quality values.

87. Q: Is integration failure fatal to detection?
	A: No. Components are designed to degrade gracefully.

88. Q: Why keep these integrations optional?
	A: To preserve robust core detection even in offline or constrained environments.

89. Q: What test evidence exists for integrations?
	A: Unit tests in tests/test_integration_pipeline.py mock external dependencies and verify payload logic and fallback behavior.

## 8) Hard Professor Challenge Questions (with strong answers)

90. Q: Is your AI actually making detections?
	A: No. Detection score originates from IDS + UEBA + fusion. AI augments by contextual reasoning and action intelligence.

91. Q: Then why call it AI-powered detection?
	A: The pipeline is hybrid: ML/anomaly detection performs scoring, and an agentic LLM layer provides explainable autonomous response orchestration.

92. Q: How do you control hallucination risk?
	A: Constrained action set, bounded tool registry, max tool calls, structured parsing, and deterministic rule fallback.

93. Q: Why local Ollama over cloud API?
	A: Cost control, data privacy, offline availability, and reproducibility during demos.

94. Q: What is your biggest technical debt today?
	A: Production-hardening of response controls and stronger benchmark-grade evaluation with ground-truth datasets.

95. Q: What is your biggest algorithmic risk today?
	A: Data drift and label mismatch between training assumptions and live cloud behavior.

96. Q: What is your biggest systems risk today?
	A: Cloud telemetry delay, particularly CloudTrail latency, which can delay UEBA signal updates.

97. Q: Why use both IDS and UEBA instead of one model?
	A: They observe different attack surfaces: traffic anomalies vs behavioral anomalies. Fusion increases robustness.

98. Q: Could simple thresholds replace all ML here?
	A: For basic spikes yes, but ML helps capture nuanced patterns and reduces manual rule maintenance.

99. Q: What is your explainability story to SOC teams?
	A: Final score decomposition (network vs user), action thresholds, stored reasoning text, and tool evidence trail.

100. Q: How do you handle concept drift?
	 A: Feedback outcomes are stored; adaptive-threshold suggestions are computed. Next step is controlled auto-update workflow.

101. Q: Is your adaptive loop online learning of model weights?
	 A: Not yet. Current adaptation is policy-threshold guidance, not direct model retraining.

102. Q: Why does your action logic use strict greater-than thresholds?
	 A: That is an implementation detail for clear bucket separation; values exactly on boundary fall to the lower severity bucket.

103. Q: Why does the diagnostics script use different proxy labels?
	 A: To provide transparent local evaluation when true labels are unavailable; it is explicitly documented as proxy analysis.

104. Q: What is your plan to improve this after demo?
	 A: Build labeled replay datasets, unify feature definitions between runtime and diagnostics, and add proper deny-capable response control.
